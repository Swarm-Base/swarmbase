00 Abstract

Single agent architectures have reached their ceiling. A single model operating in isolation lacks the depth of reasoning, the breadth of capability, and the fault tolerance required for complex real world tasks. The future of autonomous AI is not a better single agent. It is coordinated swarms of specialized agents, each contributing domain expertise, collaborating on shared objectives, and producing outputs that no individual agent could achieve alone.

SwarmBase is the infrastructure layer that makes this possible on chain. Built natively on BNB Chain, SwarmBase provides the coordination, verification, and economic framework for deploying, orchestrating, and monetizing autonomous agent swarms at scale. The protocol operates at core.swarmbase.io, with a pre-TGE contributor points system live at core.swarmbase.io/points.

This whitepaper presents the technical architecture, consensus design, cryptographic verification pipeline, tokenomics, and governance framework of the SwarmBase protocol.

01 The Swarm Thesis
1.1The Limits of Single Agent Models

The dominant paradigm in AI deployment today is the single agent model: one model, one context window, one set of capabilities, one task at a time. This architecture is fundamentally constrained in three dimensions.

Reasoning depth. A single model is bounded by its training distribution and context limits. It cannot decompose a complex problem into subtasks, delegate those subtasks to specialists, and then synthesize the results. As task complexity increases, single agent performance degrades predictably.

Capability breadth. No single model excels at everything. A model trained for code generation is suboptimal for market analysis. A model tuned for natural language reasoning may fail at structured data extraction. The single agent paradigm forces users to choose between specialized performance and general coverage, with no way to achieve both simultaneously.

Scalability. Single agent architectures do not scale horizontally. There is no mechanism for distributing cognitive load across a network of cooperating agents, each absorbing a fraction of the total workload.

1.2Swarms as the Native Architecture for Complex AI Work

A swarm is a coordinated set of specialized agents operating as a unified system. Each agent in a swarm is purpose built for a specific capability domain. A research agent retrieves and synthesizes information. An analysis agent processes structured data. A strategy agent generates recommendations. A verification agent audits outputs from other agents. These specialists are composed into directed workflows where the output of one agent feeds the input of the next, and the swarm produces results that exceed the capacity of any individual member.

The implications are significant. A swarm of five specialized 7B parameter agents, each operating in its domain of expertise, consistently outperforms a single 70B parameter generalist on complex multi-step tasks. The swarm is cheaper to run, faster to execute, and more resilient to individual component failures. When one agent fails or produces low quality output, the swarm can reroute, retry, or substitute without collapsing the entire workflow.

1.3Why Swarms Require Decentralized Infrastructure

Running swarms on centralized infrastructure recreates the same bottlenecks and trust dependencies that blockchain was designed to eliminate. A centralized swarm operator controls which agents participate, which tasks are routed where, and whether outputs are legitimate. There is no independent verification, no economic accountability, and no permissionless access.

SwarmBase decentralizes every layer of the swarm stack. Agent operators are independent participants who stake economic value against the quality of their work. Task routing is governed by on-chain logic, not a centralized dispatcher. Outputs are cryptographically verified through hardware attestation and, in the future, zero knowledge proofs. Economic rewards flow automatically to agents and validators based on measurable contributions.

02 Protocol Architecture
2.1Chain Architecture

SwarmBase is built natively on BNB Chain, leveraging both opBNB Mainnet and BNB Smart Chain (BSC) in a dual layer architecture optimized for the distinct requirements of agent execution and economic settlement.

Chain Reference

The production $SWARM token is a BEP-20 deployment on BNB Smart Chain (BSC, ChainId 56), deployed fresh at TGE. Pre-TGE engagement contracts (SwarmCore, SwarmBadge) are deployed on opBNB Mainnet (ChainId 204) permanently. A test deployment of SwarmToken exists on opBNB for pre-TGE testing only and is not the production token.

The opBNB execution layer handles all high frequency operations: agent task processing, inter-agent messaging, state transitions, and coordination logic. opBNB provides sub-second block times and transaction costs below $0.001, essential for the rapid message passing that real time swarm coordination demands. Agent state commitments are batched into Merkle roots and anchored to BSC at epoch boundaries for finality.

BSC serves as the settlement layer. Token transfers, staking, governance, and slashing execute on BSC, inheriting its validator set security. The opBNB to BSC bridge uses the native optimistic bridge with a configurable challenge period, and a relayer network monitors both chains for state inconsistencies.

2.2Swarm Composition Engine

The core primitive of SwarmBase is the swarm: a directed acyclic graph (DAG) of agent nodes connected by typed message channels. Each node is an autonomous agent with a defined capability, and the edges define the flow of data and control between agents.

Swarm topologies are defined declaratively. A user specifies which agents participate, how they connect, what data flows between them, and what conditions trigger each stage of execution. The SwarmCoordinator contract (planned, Phase 4) will manage task routing, output aggregation, fault detection, and re-routing within the swarm. Swarms can be nested — a single node in a high level swarm can itself be a sub-swarm, resolving to a single output from the perspective of the parent graph.

2.3Agent Execution Environment

Agents on SwarmBase operate within a managed execution environment at core.swarmbase.io. Each agent is configured with a defined capability set, resource allocation, and economic parameters. The agent lifecycle covers activation (entering the operational pool), execution (processing tasks and accruing rewards), and deactivation (withdrawing with staked tokens entering unbonding).

2.4State Management

Agent state is maintained in a hierarchical Merkle Patricia Trie on the opBNB execution layer. Each agent holds a local state partition that updates atomically per task execution. The global state root is committed to BSC at the end of each epoch via a propose-commit-reveal pattern: an agent proposes a state update with an attached verification proof, validators attest to its validity, and the update commits once two thirds of validator stake has confirmed.

03 Verification Pipeline
Planned — Phase 4

The verification pipeline described in this section is planned for Phase 4 delivery. No verification contracts are currently deployed.

In a decentralized swarm environment, verification is not optional. When agent operators are independent, economically motivated participants, the protocol must provide cryptographic guarantees that agent outputs are genuine products of declared computations. Without this, the entire economic model collapses: there is no basis for rewarding legitimate work or penalizing fabrication.

3.1TEE Attestation

The primary verification mechanism uses Trusted Execution Environment attestation. Agent operators run inference workloads inside hardware enclaves (Intel SGX or AMD SEV) that produce signed attestation reports, cryptographically binding the agent's output to the specific model binary that executed inside a genuine, untampered enclave.

The attestation flow: the operator initializes an enclave with the agent's model binary. The enclave generates a hardware signed report containing a measurement hash of the loaded code. This report is submitted to the VerifierGateway contract on BSC alongside the agent's output. The contract validates the hardware signature and marks the output as attested if verification passes. Only attested outputs are eligible for reward distribution.

3.2ZKML Proof System

The next evolution of the verification pipeline targets zero knowledge machine learning (ZKML) proofs using PLONK arithmetization. Agent inference computations are expressed as arithmetic circuits, and a PLONK-based zk-SNARK prover generates succinct proofs that a given output was produced by a specific model on specific inputs, without revealing model weights or raw input data.

Design targets include support for transformer architectures up to 7B parameters, GPU accelerated proving with target generation times under 30 seconds, and recursive proof composition enabling verification of entire multi-agent swarm workflows through a single aggregated proof. TEE and ZKML operate simultaneously — operators choose based on their infrastructure and security requirements.

3.3VRF Committee Selection

Validator committee composition uses a Verifiable Random Function (VRF) mechanism based on threshold BLS signatures. A committee of validators collectively produces a random beacon each epoch, and the beacon output determines the next epoch's committee composition, task routing seeds, and probabilistic protocol parameters. No participant can predict or manipulate committee assignments.

04 Proof of Inference
Planned — Phase 5

The Proof of Inference consensus layer is planned for Phase 5 delivery.

SwarmBase introduces Proof of Inference (PoI) as its consensus primitive. PoI replaces arbitrary hash computation with useful inference work. An agent's contribution to the network is measured by the verifiable inference tasks it completes, weighted by task complexity and output quality.

The PoI flow: a task is assigned to an agent via the swarm coordinator's routing schedule. The agent executes inference and produces an output bundle containing the result, a TEE attestation or ZKML proof, model identifier, input hash, and compute resource metrics. The output bundle is broadcast to the validator set. Validators independently verify the proof, evaluate output quality heuristics, and submit attestation votes. A task finalizes when two thirds of the validator stake has attested to its validity.

PoI tracks contributions at the individual agent level while evaluating outcomes at the swarm level — ensuring rewards flow to agents that actually contributed to successful executions, not just to agents that processed the most volume.

4.1Validator Requirements

Validators stake SWARM tokens to participate in the attestation process. Minimum validator stake is 50,000 SWARM (governance adjustable). Validators earn epoch emission rewards proportional to their attestation participation rate. Validators that miss attestation windows receive a liveness penalty. Validators that submit provably false attestations are subject to slashing.

5.2Emissions
Planned — Phase 3

The epoch emission and halving schedule is a planned post-TGE mechanism. It does not exist in any currently deployed contract.

Ecosystem emissions follow a halving decay. Initial annual emission: 87,500,000 SWARM (25% of the ecosystem allocation), halving every 12 months. Epoch length is 6 hours (1,460 epochs per year), yielding approximately 59,932 SWARM per epoch in year one. This front-loads incentives during the bootstrapping phase while the halving schedule ensures long term sustainability.

5.3Protocol Fees

The protocol charges 3% on agent task execution payments: 2% is burned permanently, reducing circulating supply over time; 0.5% flows to the protocol treasury; and 0.5% is distributed to validators proportional to their attestation participation. As swarm activity grows, the burn rate accelerates, creating a structural relationship between network usage and token scarcity.

5.4Staking
Planned — Phase 3

StakeVault.sol will be deployed on BSC in Phase 3.

SWARM holders stake in StakeVault to earn epoch emissions and fee distributions. Staked tokens are subject to a 14-day unbonding period. Delegated staking allows non-operator holders to delegate to a validator node and receive proportional rewards minus a configurable commission rate — ensuring all token holders, not just those running infrastructure, can participate in securing the network.

06 Pre-TGE Community Engagement
Live Now

The pre-TGE engagement system is live at core.swarmbase.io and core.swarmbase.io/points. All engagement is recorded on-chain via SwarmCore on opBNB Mainnet and publicly verifiable on opBNBscan.

6.1Earning Mechanics

Registration (one-time). 50 points awarded on first registration. Each wallet may register once. Registrations with a referral link are recorded on-chain and permanently link the referee to the referrer.

Daily check-in. Users may check in once every 24 hours. Points scale with the user's current streak: points = 25 × (100 + (streak − 1) × 7) / 100, where streak is capped at 30. This yields 25 points on day 1, rising to approximately 76 points at streak day 30. A streak resets to 1 if the user misses more than 48 hours between check-ins.

Streak milestones. Bonus points are awarded at lifetime check-in thresholds: every 7th check-in = +150 points; every 30th check-in = +500 points; every 90th check-in = +1,500 points. If a check-in triggers multiple thresholds simultaneously, only the highest applies.

Referral immediate reward. When a referred wallet registers, the referrer receives 10 points on-chain immediately.

Referral quality milestones. Additional points flow to the referrer as each referee demonstrates sustained engagement:

6.3TGE Snapshot and Airdrop

At TGE, a snapshot of all swarmScore values is taken from SwarmCore. Off-chain sybil filtering is applied; cluster analysis of referral trees, timing patterns, and funding sources is used to identify and exclude coordinated farming. The filtered score distribution is used to determine each wallet's allocation weight from the 200,000,000 community airdrop pool.

Pre-TGE scores are one input into the community allocation; final distribution also accounts for post-TGE platform activity. 10% of the community allocation (20,000,000 SWARM) is unlocked at TGE. The remaining 180,000,000 SWARM is subject to a 1-month cliff followed by 24-month linear vesting. Points do not entitle any wallet to a specific token amount. Distribution is at the team's full discretion. There is no on-chain airdrop contract.

07 Governance
Planned — Phase 3

GovernanceModule.sol will be deployed on BSC in Phase 3.

Parameter Governance covers numeric adjustments: stake thresholds, emission rates, fee percentages, and slashing penalties. These proposals require a simple majority (>50%) of participating stake-weighted votes with a minimum quorum of 10% of total staked supply.

Structural Governance covers contract upgrades, treasury disbursements, and changes to the protocol's core architecture. These proposals require a supermajority (>66.7%) with a minimum quorum of 20% of total staked supply.

7.1Proposal Flow

A governance proposal progresses through four phases: Submission — proposer locks a 10,000 SWARM bond. Discussion — 7-day community review window. Voting — 5-day window; For, Against, or Abstain; Abstain counts toward quorum but not approval. Execution — 48-hour timelock; auto-executes on-chain if not vetoed.

7.2Security Council

A 5-of-9 multisig Security Council holds emergency powers during the protocol's initial maturation phase. The council can veto governance proposals within the 48-hour timelock window and execute emergency parameter changes in response to active exploits. Council authority is designed to be progressively reduced through governance votes, targeting full dissolution within 24 months of mainnet launch.

08 Security

Fabricated outputs. Malicious agents submitting fake inference results to earn rewards. Mitigated through TEE attestation and ZKML mathematical proofs (Phase 4/5). Both mechanisms make fabrication economically irrational: the cost of bypassing hardware or mathematical verification exceeds the potential reward.

Validator collusion. Coordinated false attestations. Mitigated through stake-weighted random committee selection via VRF (Phase 5), economic penalties via slashing (up to 100% of stake for fabrication), and the two-thirds supermajority finalization threshold.

Economic attacks. Stake manipulation, MEV extraction from task routing, and oracle manipulation. Mitigated through the 14-day unbonding period, deterministic task routing algorithms resistant to front running, and on-chain governance for parameter changes.

Smart contract vulnerabilities. Reentrancy, integer overflow, access control failures. Addressed through mandatory third-party audits prior to TGE, formal verification of critical contract paths, and a tiered bug bounty program.

09 Token Contract Design

The SWARM token contract implements the full BEP-20 interface with no transfer restrictions, rebasing logic, or fee-on-transfer mechanics. The contract contains no proxy upgradeability patterns, no mint function beyond the initial supply creation, and no blacklist functions. Transfer events comply strictly with the BEP-20 specification to ensure full compatibility with block explorers, indexing infrastructure, and exchange integration pipelines.

At TGE, 80,000,000 SWARM (8% of total supply) deploys to DEX liquidity pools on BSC. LP tokens are locked via Team.Finance for a minimum of 12 months.
